# 进阶八股：从底层网络到首屏渲染的全链路解析

> **面试策略**：当面试官问到“如何优化 Web 性能”时，不要急着说代码层面的防抖节流。**先从宏观的网络请求讲起，再过渡到浏览器渲染**。这能展现你大局观和排查问题的深度。

## 一、 核心体验指标：网页加载到底有多快？

业务的成功（留存率、付费转化、PV/UV）高度依赖于页面的打开速度。前端有两个至关重要的性能里程碑：

### 1. TTFB (Time To First Byte - 首字节时间)

这衡量的是**网络和服务器的响应速度**。它涵盖了从用户点击回车，到浏览器收到服务器返回的**第一个字节**的全部时间。

* **包含的过程**：DNS 解析 -> TCP 建连 -> TLS 握手 -> 服务端处理（如查数据库、慢查询） -> 数据在网络中的回传。
* **优化切入点**：CDN 边缘节点（缩短物理距离）、HTTP/2 多路复用、优化后端查询逻辑。

### 2. FP (First Paint - 首次绘制时间)

这衡量的是**浏览器渲染引擎的工作速度**。指的是从页面加载开始，到屏幕上首次出现像素变化（比如背景色改变）的时间。

* **完整耗时公式**：
> **FP = TTFB + HTML 下载完毕 + 构建 DOM 树 + 构建 CSSOM 树 + 构建渲染树 (Render Tree) + 布局计算 (Layout) + 首次绘制 (Paint)**


* **意义**：FP 标志着页面白屏期的结束，给用户吃下第一颗“定心丸”，告诉用户“页面正在加载中”。

---

## 二、 底层支撑：一个数据包的奇妙旅程

为什么我们需要关心 TCP/IP？因为 TTFB 里的每一毫秒，都在受底层网络协议的支配。现代互联网安全的基石可以总结为：**HTTP（业务逻辑）+ TLS（防偷窥篡改）+ TCP（防丢失错乱）+ IP（精准导航）**。

### 1. 拆包与传输策略

* 由于网络带宽限制和并发需求，大文件（如 HTML/JS 资源）不可能一次性发完。
* 数据在底层会被**拆分成一个个小的数据包（数据帧）**，分批次、多通道并发传输，极大提升了带宽利用率。

### 2. IP 层（网络层）：负责“送到哪”

* 利用 IP 地址，负责在复杂的互联网中进行路由选择，将数据包从源主机导航到目标主机。
* **缺点**：IP 层只管送，不管丢包，也不管数据有没有出错。

## 三、 传输层的抉择：UDP 还是 TCP？

数据包送达目标机器后，还需要交给正确的应用（通过**端口号**识别）。此时面临两种传输协议的选择。

- 网络层（IP）：只负责把数据送到某一台机器（通过 IP 地址）。
- 一台机器上会跑很多程序（浏览器、游戏、音乐等），所以还要决定：把数据交给哪一个程序。传输层就是干这件事的：用端口号区分不同应用，把数据交给“对的进程”。
- 传输层有两种实现方式：TCP 或 UDP。

| 对比维度 | UDP (用户数据报协议) | TCP (传输控制协议) |
| --- | --- | --- |
| **核心特点** | 简单、快速、无连接。 | 可靠、面向连接、稍慢。 |
| **对待数据** | 发出去就不管了，丢包不重传，乱序不重排。 | 丢包重传、乱序重组、严格确认。 |
| **适用场景** | 音视频直播、在线游戏 (偶尔卡顿/掉帧可接受)。 | **Web 资源加载 (HTML/CSS/JS)、邮件**。 |

**为什么 Web 必须用 TCP？**
因为 HTML 这种结构化代码极其脆弱，少了一个闭合标签 `</div>` 或者一段 JS 脚本顺序错乱，整个页面就会崩溃。TCP 完美解决了 UDP 的痛点：

1. **解决丢包**：设置过期时间，一旦没收到确认 (ACK)，立马**重传**。
2. **解决乱序**：给每个数据包打上**序号**，接收端按顺序**重组**。

## 四、 面试杀手锏：TCP 握手与挥手的本质逻辑

很多时候，我们只记得“三次握手，四次挥手”，但如果面试官问“**为什么必须是三次？**”，你可以用你笔记里的这段**绝佳逻辑**来回答：

### 深刻理解“三次握手”

通信的本质是确认**双方的发送和接收能力**。TCP 是全双工的，理论上需要 4 次确认（各自确认一发一收），但 TCP 巧妙地将中间两次合并了。

1. **第一次（客 -> 服 发 SYN）**：
* 动作发生后：**服务端知道**了客户端能发数据。


2. **第二次（服 -> 客 发 SYN + ACK）**：
* 动作发生后：**客户端知道**了服务端能收（因为收到了针对自己 SYN 的 ACK），也知道了服务端能发（因为收到了服务端的 SYN）。
* *此时，客户端的视角已经圆满了（知道自己能发能收，也知道对方能发能收），但服务端的视角还缺一块。*


3. **第三次（客 -> 服 发 ACK）**：
* 动作发生后：**服务端知道**了客户端能收（因为收到了针对自己 SYN 的 ACK）。至此，双方的收发能力都确认完毕，正式建立连接。



### 简述“四次挥手”

因为是全双工，双方都要独立关闭自己的发送通道。

* 客户端发 `FIN`（我说完了）-> 服务端回 `ACK`（收到，但我可能还有话没说完，你等会）。
* 服务端处理完剩余数据，发 `FIN`（我也说完了）-> 客户端回 `ACK`（彻底拜拜）。
